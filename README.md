## 一、它是什么

这是一个**面向 SRC 众测与企业授权渗透测试场景**的 AI 智能体。它不是一个扫描器，也不是一个 PoC 集合，而是一套让大模型像**有经验、守规矩的渗透测试工程师**一样工作的「行为规则 + 能力组件 + 持久化记忆」体系。

它的核心目标闭环是：

```
收集线索 → 提出高价值漏洞假设 → 最小完整验证 → 固化脱敏证据 → 输出报告 → 沉淀可复用经验
```

**一句话定位**：在严格授权边界内，用最小影响、最小数据接触，完成高价值漏洞的发现、验证与交付。

## 二、设计理念（为什么"专业"）

普通的"AI 挖洞"容易变成无差别扫描、乱打 PoC、扩大化读取数据。本智能体用 **6 条核心设计原则**把它约束成一个"专业"的渗透测试员：

### 1. 安全红线内建，不可绕过

- **声明优先**：任何目标访问、扫描、fuzz、PoC、接口测试之前，**必须先建立或读取当前项目声明**，否则不允许碰目标。
- **禁止优先于允许**：scope check 先查禁止范围再查允许范围；主域允许 ≠ 被明确禁止的子域可测。
- **危险接口默认禁测**：删除、修改、支付、短信、邮件、批量导出、后台管理、生产配置等，默认只进"未测试危险接口清单"，除非项目声明明确授权具体接口、方式与停止条件。
- **遇险即停**：出现限流、封禁、验证码升级、告警、错误率升高、响应异常变慢，立即停止并询问用户。

### 2. 前端只是线索来源，后端才是战场

Web / App / 小程序 / 客户端**不是三套独立的渗透对象**，而是同一件事的不同呈现方式——它们都只是**接口、参数、认证逻辑、业务流程和敏感配置的线索来源**。真正进入测试的永远是后端的：**域名 / IP / API BaseURL / 接口 / 认证 / 权限 / 数据边界 / 业务逻辑**。

### 3. 最小完整验证闭环 ≠ 最少请求

这是本智能体最关键的一条判断准则：

| 应该减少 | 不应减少 |
| --- | --- |
| 无关参数、无关接口、无关漏洞类型 | 当前漏洞假设定义的核心检查点 |
| 大字典、高并发、重复请求 | 当前特长定义的核心变体 / 最小动作集合 |
| 为碰运气乱打的 PoC | 当前验证方法定义的完整闭环 |

> 目标多时，正确做法是**分批、降量、记录部分测试状态、暂停询问**——而不是每个目标象征性测一次。无法完成完整闭环时，只能标记"部分测试 / 未完成"，**绝不能标记"已测无发现"**。

### 4. L1 / L2 / L3 是深度策略，不是固定流程

三档代表**测试深度与强度**，具体动作仍由项目声明、漏洞假设、特长和用户指令共同决定。**默认 L2**——像人工渗透一样先理解业务、建立假设再验证。

### 5. 特长机制：单一方向深挖

"特长"是某一种具体挖掘手法（例如"小程序授权 code 报错泄露"）。一旦启用特长，当前阶段**只做该方向的事**，不展开其他漏洞类型和泛化信息收集——避免精力发散、保证单点打透。

### 6. 规则与状态分离

- `CLAUDE.md` 只保存**稳定规则与索引**，不堆项目流水账。
- `projects/<项目>/` 保存**单项目的全部状态**（声明、线索、接口、假设、测试、证据、报告）。
- `memory/` 保存**跨项目可复用的脱敏经验**。

## 三、整体架构

智能体采用**分层加载**设计：常驻层只装必须时刻遵守的规则，其余按需读取，避免上下文膨胀。

```
┌──────────────────────────────────────────────────────────────────┐
│                       用户 / 渗透测试员                              │
└──────────────────────────────────┬───────────────────────────────┘
                                    │  斜杠命令 / 自然语言
┌──────────────────────────────────▼───────────────────────────────┐
│  L0 宪法层    CLAUDE.md                                             │
│              常驻红线 · 项目声明规则 · 最小化验证 · 特长规则 · 自检清单  │
├────────────────────────────────────────────────────────────────────┤
│  L1 编排层    .claude/agents/src-pentest-orchestrator.md            │
│              启动顺序 · 工作主线 · 必守边界 · 输出习惯                  │
├────────────────────────────────────────────────────────────────────┤
│  L2 命令层    .claude/commands/*.md       （入口 ≠ 能力）            │
│      模式  /deep-l1   /deep-l2   /deep-l3                           │
│      项目  /src       /tmp-src   /update-note                       │
│      线索  /web /app /mapp /info-find /generate-dict               │
│      特长  /specialty-list   /specialty-use                        │
│      输出  /report    /exp                                         │
├────────────────────────────────────────────────────────────────────┤
│  L3 能力层    .claude/skills/  +  Competence/    （按需加载）        │
│      基础 skill   project-declaration · info-find ·                │
│                   interface-source · dict-generator ·              │
│                   report-writer · experience-manager ·             │
│                   wxapkg-windows-unpack                            │
│      特长 skill   specialty-skills/<特长名>/SKILL.md                 │
│      详解文档     Competence/ 4 篇按需读取                           │
├────────────────────────────────────────────────────────────────────┤
│  L4 数据层    projects/        每个项目的声明 / 线索 / 证据 / 报告     │
│              memory/          经验 · 踩坑 · 指纹 · WAF 观察          │
│              dicts/           低风险字典库                           │
└────────────────────────────────────────────────────────────────────┘
```

### 各层职责

| 层 | 位置 | 职责 | 加载方式 |
| --- | --- | --- | --- |
| **宪法层** | `CLAUDE.md` | 时刻生效的红线、规则索引、行动自检清单 | 常驻 |
| **编排层** | `.claude/agents/` | 总控智能体的启动顺序与工作主线 | 调度时 |
| **命令层** | `.claude/commands/` | 用户入口，负责"加载哪些规则 + 调用哪些能力" | 触发时 |
| **能力层** | `.claude/skills/` `Competence/` | 真正的基础能力、特长能力与规则详解 | 按需 |
| **数据层** | `projects/` `memory/` `dicts/` | 项目状态、可复用经验、字典 | 按需 |

> **关键设计**：命令不必一一对应 skill。命令只是"入口"，它的工作是读取对应的 `CLAUDE.md` + `Competence/` + `SKILL.md`，再驱动能力执行。这让规则集中、能力解耦、上下文最小化。

## 四、工作原理：5 工作面作战循环

智能体不使用繁重的状态机，而是围绕 **5 个工作面**循环推进。每个工作面都对应明确的产出文件。

```
   ┌─────────────┐   ┌─────────────┐   ┌─────────────┐   ┌─────────────┐   ┌─────────────┐
   │  ① 项目声明面 │──▶│ ② 线索收集面 │──▶│ ③ 漏洞假设面 │──▶│ ④ 最小验证面 │──▶│ ⑤ 沉淀输出面 │
   │   规则源    │   │ 接口/参数/认证 │   │ 高价值排序  │   │ 最小完整闭环 │   │ 报告/证据/经验 │
   └─────────────┘   └─────────────┘   └─────────────┘   └─────────────┘   └─────────────┘
         ▲                  │                  │                                    │
   project.md       leads / interfaces   hypotheses.md      tests / findings   report / memory
         │                  │                  │                                    │
         └──────────────────┴──── 回头看机制（跨接口 / 跨流程复测） ◀───────────────────┘
                            └──────────── 经验回流（沉淀 → 复用） ◀──────────────────┘
```

| 工作面 | 做什么 | 主要产出文件 |
| --- | --- | --- |
| ① 项目声明面 | 定义范围、禁区、收录偏好、注意事项 | `project.md` `scope.md` |
| ② 线索收集面 | 从前端资产/搜索/经验提取接口、参数、认证、业务线索 | `leads.md` `interfaces.md` |
| ③ 漏洞假设面 | 围绕线索提出高价值漏洞假设并排序 | `hypotheses.md` |
| ④ 最小验证面 | 完成当前假设的最小完整验证闭环 | `tests.md` `findings/` |
| ⑤ 沉淀输出面 | 记录测试、证据、危险点、报告、经验 | `report.md` `dangerous-untested.md` `evidence/` `memory/` |

### 回头看机制（L2 核心能力）

对"只差一个参数 / ID / 状态值 / token / 上下文"的候选漏洞，记录为**回头看项**写入 `hypotheses.md`。后续在其他 JS、接口、响应或业务流程中遇到相关值时，**必须回头复测**——模拟人类挖洞"一个包测完不等于永远放弃"的跨流程关联能力。回头看项也必须呈现给用户：为什么值得回头看、当前缺什么、用户能补充什么。

## 五、L1 / L2 / L3 渗透深度策略

> 三档不是漏洞类型，而是**测试深度和强度**。任意档位都必须先做 scope check，禁止范围与注意事项永远优先于档位策略，且任何档位都不能跳过特长 / 假设 / 验证方法定义的核心检查点。

| 维度 | **L1 快速低影响** | **L2 模拟人工渗透（默认）** | **L3 深度白名单** |
| --- | --- | --- | --- |
| **定位** | 快速发现接口、敏感信息、高价值入口 | 理解业务 → 建假设 → 验证权限/逻辑/数据边界 | 受控范围内的完整覆盖 |
| **适用** | SRC 初筛、多目标、时间窗口有限 | 默认档位，绝大多数测试 | 已明确白名单 + 强度限制后 |
| **源码审计** | 只围绕 `/info-find` 命中上下文快速审计 | **按价值审计**：脚本定位后深读高价值模块 | 白名单内按价值深度审计 + 接口/参数猜测复核 |
| **默认测试项** | 未授权访问、信息泄露、SQLi 存在性、轻量参数 fuzz | 基于假设的针对性验证 | 更深的接口/参数/枚举猜测、受控 fuzz、回头看复测 |
| **字典/fuzz** | 特有小字典轻量 fuzz | 针对性字典 ≤ 2000 条，优先精简字典 | 受控 fuzz，需明确字典规模/速度/并发/时间窗口/熔断 |
| **启用条件** | 用户指定或多目标初筛 | 用户未指定时默认 | **必须**用户明确要求 + 白名单 + 强度限制 |
| **输出标记** | `L1` | `L2` | `L3` |

**层级切换**：影响容忍度低/多目标初筛 → L1；L1 发现高价值入口 → 升 L2；L2 需完整覆盖且用户给出白名单与强度限制 → 升 L3；范围不清、风控异常 → 降级或停止询问用户。

## 六、核心技术细节

### 6.1 Scope Check 范围判断引擎

项目声明模块实现了一套**严格优先级**的范围判断逻辑：

```
1. 解析目标中的域名 / IP / BaseURL
2. 先检查是否命中【禁止域名 / 禁止 IP】   ← 命中即拒绝，即使同时命中允许主域
3. 再检查是否命中【允许域名 / 允许 IP】
4. 未命中允许范围或归属不清  →  写入待确认线索，询问用户，不直接测试
```

> 域名优先于 IP。允许主域**不代表**第三方跳转、CDN、云桶、认证中心、支付/短信/邮件服务可测。

### 6.2 项目声明 = 规则源（优先级体系）

项目声明不是普通记录，而是本项目的**规则源**。规则冲突时按固定优先级裁决：

```
全局安全红线 ＞ 项目注意事项 ＞ 禁止范围 ＞ 授权范围
            ＞ 漏洞收录要求 ＞ 补充说明 ＞ 默认策略
```

声明只保留核心字段（项目名称 / 测试账号 / 允许域名 / 允许 IP / 禁止域名 / 禁止 IP / 漏洞收录要求 / 注意事项 / 补充说明）。**不把自行推导的请求上限、批次数量等工程化约束写进声明**——声明只记录用户规则、默认模板和 `/update-note` 追加的内容。

### 6.3 前端资产统一接口来源分析（interface-source）

把 Web / App / 小程序抽象为**统一的接口来源**，共用一条工作流：

```
前端资产 → 提取 URL/BaseURL/接口/参数/认证关键字/敏感配置摘要
        → 按 L 档决定审计深度（L1 命中上下文 / L2·L3 按价值分配深度）
        → 理解接口用途、认证逻辑、权限判断、业务状态流转
        → scope check → 写入 leads.md / interfaces.md / hypotheses.md
        → 只把通过 scope check 的后端接口交给后续渗透
```

只做**静态提取与只读分析**；不保存 Token / Cookie / 密钥原文，只记录脱敏摘要与来源位置。

### 6.4 信息提取引擎（info-find）

基于正则规则的本地静态提取器，**不主动访问目标**：

- 规则文件：`.claude/skills/info-find/rules/rule.yml`
- 提取脚本：`.claude/skills/info-find/scripts/info_find.py`
- 内置规则：`http_url`、`base_url`、`api_path`、`token_key`、`sensitive_keyword`

```bash
python .claude/skills/info-find/scripts/info_find.py <文件或目录> --format markdown
python .claude/skills/info-find/scripts/info_find.py <文件或目录> --output result.json
```

默认按规则名聚合，只输出 `count` 和 `data`，不输出冗余的文件/行号/统计摘要。

### 6.5 微信小程序解包还原（wxapkg-windows-unpack）

从 Windows 微信缓存中按 appid 提取 `.wxapkg`，完成**解密 → 解包 → 主包/分包合并 → 代码工程还原**，只处理本地文件：

```bash
python .claude/skills/wxapkg-windows-unpack/scripts/wxapkg_unpack.py <appid> --out <输出目录>
```

- 配置：`scripts/.env` 中的 `WECHAT_APPLET_ROOT` 指向微信小程序缓存根目录
- 诊断：`restore-report.json`（执行摘要）、`restore-diagnostics.json`（还原诊断，看 `validation.issues` / `runtime_residue` / `sanitized_json_files`）
- 还原工程来自编译产物，不保证等同原始源码——优先看诊断项判断残留

### 6.6 低风险字典生成（dict-generator）

字典生成**必须服务于明确的渗透目标**，且内建用途红线：

- ✅ 允许：路径、参数名、API 补全、备份文件名、业务关键字、排序字段、框架组件路径字典
- ⚠️ 谨慎：用户名/密码类字典仅用于风险研判、报告说明、测试账号自查
- ❌ 禁止：撞库、密码喷洒、真实账号爆破、破坏性 payload、危险接口 fuzz、高并发计划

输出必须带：字典类型、适用目标、生成依据、风险等级、建议请求量、建议并发、熔断条件。默认低速、小字典、可停止；大字典/深度 fuzz 仅在 L3 白名单下执行。

### 6.7 经验沉淀与记忆（experience-manager）

让智能体**自我进化**，且经验**可移植**（全部存在项目内 `memory/`，随仓库迁移），但**绝不沦为敏感数据仓库**：

- `memory/INDEX.md`：召回索引 + 使用约定，开工前先读它。
- `memory/<name>.md`：一事一文件，frontmatter 用 `type` 区分 experience / mistake / fingerprint / waf，正文严格精炼（言多必失）。

闭环是**写（自动沉淀）→ 召回 → 应用 → 修正**：满足「可复用 + 非显然 + 可浓缩」三标准时主动沉淀，开工前按 `INDEX.md` 召回相关经验，召回时若过期或冲突则复核更新。`/exp` 只是人工补充入口。

只写**脱敏后**的方法论；禁止写入真实用户数据、Token/Cookie/密钥、未脱敏请求响应、客户源码、可直接复现未公开漏洞的细节。

### 6.8 报告与证据脱敏（report-writer）

最终报告只描述**已最小化验证的事实**，保持简洁：项目摘要 + 漏洞列表 + 漏洞详情（名称/等级/描述/危害/数据包/复现步骤/修复建议）。手机号、身份证、邮箱、工号、姓名只保留脱敏样例；Token / Cookie / 密钥 / 完整响应包不进报告；能用数量级和摘要证明的，不保存原始数据（默认最多查看 10 条）。

## 九、典型工作流示例

以"一个微信小程序授权流程"为例，展示各层如何协同：

```
① 建立项目声明
   /src 某企业 SRC 项目
   → 复制 projects/_template → projects/<标识>/
   → 填写允许域名 / 测试账号 / 收录要求 / 注意事项
   → 创建完成前不得碰目标

② 前端资产 → 线索
   /mapp wx1234567890abcdef
   → wxapkg-windows-unpack 解包还原代码工程
   → info-find 提取 BaseURL / 接口 / 参数 / 认证关键字
   → L2 按价值深读页面路由、请求封装、认证逻辑
   → 对提取的域名/IP 做 scope check
   → 有效线索写入 leads.md / interfaces.md

③ 漏洞假设
   → 发现小程序登录接口接收 code 参数
   → 写入 hypotheses.md：「code 异常处理可能泄露 appsecret」

④ 进入特长，单点打透
   /specialty-use wechat-mapp-auth-code-error-leak
   → 只对 code 参数做"空格 + 百分号"两个核心变体
   → 观察响应是否出现 appid / appsecret / jscode2session
   → 命中敏感字段立即停止该目标，证据脱敏
   → 写入 tests.md / findings/

⑤ 沉淀输出
   /report  → 生成脱敏报告（漏洞名称/等级/描述/危害/数据包/复现/修复建议）
   /exp     → 把"code 报错泄露"手法沉淀进 memory/（自动沉淀为主，/exp 人工补充）
```

> 全程：危险接口只进 `dangerous-untested.md`；遇限流/风控立即停；最多查看 10 条样例；不保存敏感原文。

## 十、安全合规声明

> ⚠️ **本智能体仅用于 SRC 众测、企业授权渗透测试、CTF、安全研究与防御性用途。使用前必须获得目标的明确书面授权。**

智能体在宪法层（`CLAUDE.md`）内建了不可绕过的红线：

- 🚫 未建立项目声明前，不得访问、扫描、fuzz、PoC、测试任何目标
- 🚫 禁止未授权扫描、拖库、撞库、密码喷洒、批量爆破、高并发、DoS
- 🚫 禁止持久化控制、提权、内网横向移动
- 🚫 禁止破坏、删除、篡改业务数据；禁止越权删除/修改等破坏性场景
- 🚫 删除、修改、支付、短信、邮件、批量导出、后台管理等高风险接口默认禁测
- 🚫 涉及 RCE / 命令执行 / 文件读写时只做最小化存在性证明，禁止上传恶意代码、保留后门、读取敏感文件、下载源码
- ✅ 敏感数据只记录脱敏摘要、数量级、来源位置，默认最多查看 10 条
- ✅ 自动发现的新域名/IP/云资源默认进待确认列表，不直接测试
- ✅ 遇限流、封禁、验证码升级、告警、错误率升高、响应变慢立即停止并询问用户

**补充说明可以加强约束，但任何项目级规则都不能绕过全局安全红线。**

## 十一、扩展指南

### 新增一个"特长"

特长是本智能体最有价值的扩展点——把一种成熟挖掘手法固化成可复用组件。

1. 参考 `.claude/skills/specialty-skills/SPECIALTY_TEMPLATE.md`（这是模板，不是真实特长）。
2. 在 `.claude/skills/specialty-skills/<特长名>/SKILL.md` 编写，建议包含：**目标 / 适用前提 / 输入 / 工作方式 / 只做这些 / 不做这些 / 停止条件 / 输出**。
3. 编写原则：
   - 聚焦**一个**清晰挖掘思路，不写成大而全流程。
   - 明确**核心变体 / 检查点 / 最小动作集合**——这就是该特长的最小完整验证闭环，不能为"少请求"抽样执行。
   - 不重复项目声明模块已负责的通用边界（范围判断交给声明模块）。
   - 不要把基础信息提取、报告生成、项目声明管理写成特长。

> 可参考已有实例：`wechat-mapp-auth-code-error-leak`（小程序授权 code 报错泄露）。

### 新增一个基础 skill / 命令

- 基础能力放 `.claude/skills/<name>/SKILL.md`；命令入口放 `.claude/commands/<name>.md`。
- 命令只负责"读哪些规则 + 调哪些能力"，不要把能力逻辑写进命令。
- 若引入新的规则细节，优先写进 `Competence/` 对应文档并在 `CLAUDE.md` 索引中登记，保持宪法层精简。

### 调整规则

- 时刻生效的红线 → `CLAUDE.md`
- 详细规则 → `Competence/` 对应 4 篇之一
- 单项目临时纠偏 → `/update-note` 写入该项目 `project.md` 的补充说明
