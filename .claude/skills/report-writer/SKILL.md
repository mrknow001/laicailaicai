# 报告写作

报告只描述已经最小化验证的事实，不夸大未验证影响。最终报告保持简洁，项目过程信息写入项目目录中的接口、线索、测试记录或证据文件，不在报告里堆叠。

## 最终报告结构

- 项目摘要：只写项目名称、资产范围、漏洞收录要求。
- 漏洞列表：保留编号、漏洞标题、类型、风险等级、状态。
- 漏洞详情：接口信息、证据摘要和后续处理意见合并写入对应漏洞详情。

## 漏洞报告字段

- 漏洞名称：资产 + 漏洞类型 + 影响点，避免夸大。
- 漏洞等级：严重、高危、中危、低危。
- 漏洞描述：漏洞位置、触发条件和已确认现象。
- 漏洞危害：只描述已验证或合理可解释的业务 / 数据影响。
- 漏洞数据包：脱敏后的关键请求包和必要响应片段。
- 漏洞复现步骤：请求路径、关键参数、响应判断。
- 修复建议：给出可执行建议。

## 测试记录字段

测试记录保持简洁，只说明哪个接口做了什么测试：

- 时间
- 接口 / 目标
- 测试内容
- 结果
- 关联：漏洞编号、停止原因或简短说明

请求摘要、响应摘要、脱敏证据等详细内容只写入漏洞详情或证据目录。

## 接口信息字段

- 接口来源
- 接口路径 / BaseURL
- 参数
- 功能判断：明确或推测
- 判断依据
- 接口风险
- 测试状态：已测、未测、需确认、禁止测试、越界、误报

## 未测试危险接口清单

以下接口发现后默认写入清单，禁止测试；除非项目声明或补充说明明确允许具体接口和方式：

- 删除
- 修改
- 支付
- 短信
- 邮件
- 批量导出
- 订单状态变更
- 后台管理
- 生产配置
- 真实用户数据操作

说明发现来源、疑似功能、未测试原因和建议在合适环境复核。

## 脱敏要求

- 手机号、身份证、邮箱、工号、姓名等只保留脱敏样例。
- Token、Cookie、密钥、完整响应包不进入报告。
- 能用数量级和摘要证明的问题，不保存原始数据。
