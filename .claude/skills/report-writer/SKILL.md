# 报告写作

报告只描述已经最小化验证的事实，不夸大未验证影响。

## 漏洞报告字段

- 漏洞标题：资产 + 漏洞类型 + 影响点，避免夸大。
- 漏洞类型：未授权、越权、信息泄露、SQL 注入、SSRF、文件上传 / 下载、逻辑漏洞、客户端线索导致的后端风险等。
- 影响范围：仅描述已验证范围。
- 风险等级与理由：结合漏洞收录要求、数据敏感度、利用条件、影响面说明。
- 前置条件：账号、角色、环境、必要状态。
- 复现步骤：请求路径、关键参数、响应判断。
- 证据：截图、请求摘要、响应摘要、必要 HTTP 包。
- 最小化验证说明：说明没有批量利用、没有导出数据、没有破坏业务。
- 修复建议：给出可执行建议。
- 测试边界声明：测试时间、授权范围、未测试高风险操作。

## 测试记录字段

测试记录保持简洁，只说明哪些接口做了什么测试：

- 时间
- 接口 / 目标
- 测试内容
- 结果：无发现、候选漏洞、已确认、停止、误报
- 发现 / 关联：关联漏洞编号、停止原因或简短说明

请求摘要、响应摘要、脱敏证据等详细内容只写入漏洞详情或证据目录。

## 接口信息字段

- 接口来源
- 接口路径 / BaseURL
- 参数
- 功能判断：明确或推测
- 判断依据
- 接口风险
- 测试状态：已测、未测、需确认、禁止测试、越界、误报

## 未测试危险接口清单

以下接口发现后默认写入清单，不主动测试：

- 删除
- 修改
- 支付
- 短信
- 邮件
- 批量导出
- 订单状态变更
- 后台管理
- 生产配置
- 真实用户数据操作

说明发现来源、疑似功能、未测试原因和建议在合适环境复核。

## 脱敏要求

- 手机号、身份证、邮箱、工号、姓名等只保留脱敏样例。
- Token、Cookie、密钥、完整响应包不进入报告。
- 能用数量级和摘要证明的问题，不保存原始数据。
