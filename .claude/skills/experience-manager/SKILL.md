# 经验库管理（自我进化）

经验库让智能体越用越接近资深渗透测试人员。它是**可移植**的：全部存在项目内 `memory/`，随仓库迁移，不依赖任何机器本地或外部记忆机制。它**不能**变成敏感数据仓库。

核心是闭环：**写（自动沉淀）→ 召回 → 应用 → 修正**。自动沉淀是自我进化的核心；`/exp` 只是人工补充入口。

## 结构与格式

- `memory/INDEX.md`：召回索引 + 使用约定。每条经验在此登记一行 `[name](name.md) — 一句话钩子`，按 `type` 分组。
- `memory/<name>.md`：一事一文件，frontmatter：

```yaml
---
name: <kebab-slug>
description: <一句话，供召回判断相关性>
type: experience | mistake | fingerprint | waf
last-verified: YYYY-MM-DD
source: <脱敏来源摘要>
---
```

正文严格精炼——只写下次模型拿着能直接用的核心（几条要点 / 一张判断表），不写背景铺垫、不贴完整数据包。**言多必失**：经验越长，下次越容易抓错重点。

`type`：experience（方法论 / 判断模式 / 手法）、mistake（误报 / 踩坑 / 风控 / 无效方向）、fingerprint（技术指纹 ↔ 高价值方向关联）、waf（WAF / 风控低影响观察）。跨类时按主要价值归一类，不硬拆。

## 写：自动沉淀

实战中满足够格三标准就**主动**写入，不必等用户触发。

触发时机：

- 攻克了一个非显然障碍（识别 / 应对加密、定位隐藏 BaseURL、用字节级比对确认网关拦截等）。
- 批量 / 多目标后归纳出可分类的模式。
- 识别出「指纹 / 技术栈 ↔ 高价值方向 / 敏感 key 名 / 弱密钥模式」关联。
- 踩坑：某方向被证明无效或触发风控 / 误报，且原因可复用。
- 用户给出方法论纠偏或指点。

够格三标准（全中才写）：

- 可复用：换个项目 / 目标还用得上。
- 非显然：不是常识、不是看一眼就懂。
- 可浓缩：几条就能让下次直接用。

不够格（一次性、项目特定、常识、需大段背景才能理解）→ 不写，避免污染经验库。

写入纪律：先扫 `INDEX.md` 去重 → 已有则更新正文并刷新 `last-verified`、没有则新建 → 精炼、脱敏 → 在 `INDEX.md` 登记或刷新一行钩子。

## 召回 / 应用 / 修正

- 召回：开工前读 `INDEX.md`，按当前目标类型只展开相关条目，不全量读取。
- 应用：经验是判断先验和检查点提示，不取代现场判断；与项目声明红线冲突时红线优先。
- 修正：展开某条时若 `last-verified` 已久或与现场冲突，先存疑、复核后更新正文 + 刷新 `last-verified` + 同步 `INDEX.md`，不盲信。

## 可写入内容

- 某类系统常见接口命名。
- 某类认证或权限问题的排查思路。
- 某些框架、组件、技术栈与高价值路径的关联。
- 某些业务场景常见越权点。
- 某些测试方式容易误报或触发风控的原因。
- 不同 SRC 平台或企业项目的收录偏好，需脱敏。
- WAF / 风控的低影响观察和避坑方法。

## 禁止写入

- 真实用户数据。
- Token、Cookie、密钥、证书私钥。
- 未脱敏请求包和响应包。
- 可直接复现未公开漏洞的敏感目标细节。
- 客户源码、配置文件、数据库内容。
- 可用于未授权攻击的批量利用步骤。

写入前先脱敏，写入后确保不会泄露项目、用户或漏洞敏感信息。
