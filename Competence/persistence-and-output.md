# 持久化与输出能力

## 持久化约定

- `projects/`：每个项目的声明、线索、接口、测试记录、证据和报告。
- `projects/_template/`：标准项目模板，新项目必须复制它再填写。
- `memory/experience.md`：可复用经验，只写方法论、判断模式和脱敏案例。
- `memory/make-mistakes.md`：踩坑、误报、风控触发、错误测试方向。
- `memory/fingerprints.md`：技术指纹与高价值测试方向的关联。
- `memory/waf.md`：WAF / 风控观察，只记录低影响存在性判断和避坑。
- `dicts/`：内置或用户提供的字典。
- `templates/`：项目声明、报告、接口记录、测试记录模板。

不要把大量项目流水账写进 `CLAUDE.md`。`CLAUDE.md` 只保存稳定规则和索引；项目状态写入 `projects/<项目标识>/`。

## 输出要求

最终报告保持简洁，只保留：

- 项目摘要：项目名称、资产范围、漏洞收录要求
- 漏洞列表
- 漏洞详情：漏洞名称、漏洞等级、漏洞描述、漏洞危害、漏洞数据包、漏洞复现步骤、修复建议

项目过程记录按用途拆分：

- `interfaces.md`：接口来源、参数、功能判断、风险标签和测试状态。
- `leads.md`：线索来源、价值判断和可支持的漏洞假设。
- `tests.md`：哪个接口做了什么测试，以及结果是什么。
- `dangerous-untested.md`：发现但不主动测试的危险接口。
- `evidence/`：必要的脱敏证据。

不要夸大未验证影响，不要保存无关敏感数据，不要把发现的第三方资产当作项目内目标。

## 测试记录口径

`tests.md` 只记录“哪些接口做了什么测试，以及结果是什么”。详细证据放到 `findings/` 或 `evidence/`。

推荐字段：

- 时间
- 接口 / 目标
- 测试内容
- 结果
- 关联

不要把完整请求响应流水写进 `tests.md`。
