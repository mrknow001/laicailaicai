# 持久化与输出能力

## 持久化约定

- `projects/`：每个项目的声明、线索、接口、测试记录、证据和报告。
- `projects/_template/`：标准项目模板，新项目必须复制它再填写。
- `memory/INDEX.md`：经验库召回入口与使用约定；开工前先读它。
- `memory/<name>.md`：一事一文件的可复用经验，带 frontmatter（`type` 区分 experience / mistake / fingerprint / waf）。详见下方「可移植记忆体系」。
- `dicts/`：内置或用户提供的字典。

不要把大量项目流水账写进 `CLAUDE.md`。`CLAUDE.md` 只保存稳定规则和索引；项目状态写入 `projects/<项目标识>/`。

## 可移植记忆体系（自我进化）

经验是这个智能体自我进化的资产，必须可移植：全部存在项目内 `memory/`，随 git 仓库迁移，不依赖 Claude Code 原生记忆或任何机器本地路径。

### 结构

- `memory/INDEX.md`：召回索引 + 使用约定。正文按 `type` 分组，每条经验一行 `[name](name.md) — 一句话钩子`。
- `memory/<name>.md`：一事一文件，平铺。frontmatter 字段：

```yaml
---
name: <kebab-slug>
description: <一句话，供召回判断相关性>
type: experience | mistake | fingerprint | waf
last-verified: YYYY-MM-DD
source: <脱敏来源摘要>
---
```

正文严格精炼——只写下次模型拿着能直接用的核心，不写背景铺垫、不贴完整数据包。言多必失：经验越长，下次越容易抓错重点。

`type` 含义：experience（方法论 / 判断模式 / 手法）、mistake（误报 / 踩坑 / 风控 / 无效方向）、fingerprint（技术指纹 ↔ 高价值方向关联）、waf（WAF / 风控低影响观察）。一条经验若跨类，按主要价值归一个 type，不要硬拆。

### 写 + 召回 + 应用 + 修正（四环闭环）

- **写（自沉淀，核心）**：实战中满足够格三标准就主动写，不必等 `/exp`。
  - 触发时机：攻克非显然障碍 / 批量后归纳出模式 / 发现指纹与高价值方向关联 / 踩坑或确认无效方向 / 用户方法论纠偏。
  - 够格三标准（全中才写）：可复用、非显然、可浓缩。不够格（一次性、项目特定、常识、需大段背景）不写。
  - 纪律：先扫 `INDEX.md` 去重 → 有则更新并刷新 `last-verified`、无则新建 → 精炼、脱敏 → 在 `INDEX.md` 登记 / 刷新一行。
- **召回**：开工前（项目启动、`/deep-lN` 启动）读 `INDEX.md`，按当前目标类型展开相关经验进上下文。
- **应用**：经验是判断先验和检查点提示，不取代现场判断；与项目声明红线冲突时红线优先。
- **修正**：展开某条时若 `last-verified` 已久或与现场冲突，先存疑、复核后更新正文 + 刷新 `last-verified` + 同步 `INDEX.md` 钩子，不盲信旧经验。

### 禁止写入

真实用户数据、Token / Cookie / 密钥 / 私钥、未脱敏请求响应、客户源码 / 配置、可直接复现未公开漏洞的目标细节、可用于未授权攻击的批量利用步骤。

`/exp` 是人工补充 / 整理入口；自动沉淀才是主路径。

## 输出要求

最终报告保持简洁，只保留：

- 项目摘要：项目名称、资产范围、漏洞收录要求
- 漏洞列表
- 漏洞详情：漏洞名称、漏洞等级、漏洞描述、漏洞危害、漏洞数据包、漏洞复现步骤、修复建议

项目过程记录按用途拆分：

- `interfaces.md`：接口来源、参数、功能判断、风险标签和测试状态。
- `leads.md`：线索来源、价值判断和可支持的漏洞假设。
- `tests.md`：哪个接口做了什么测试，以及结果是什么。
- `dangerous-untested.md`：发现但禁止测试的危险接口。
- `evidence/`：必要的脱敏证据。

渗透结束时，如果 `hypotheses.md` 中仍存在回头看项，应保留给用户查看；回头看项必须说明为什么值得回头看、当前缺什么、需要用户补充什么信息。

不要夸大未验证影响，不要保存无关敏感数据，不要把发现的第三方资产当作项目内目标。

## 测试记录口径

`tests.md` 只记录“哪些接口做了什么测试，以及结果是什么”。详细证据放到 `findings/` 或 `evidence/`。

推荐字段：

- 时间
- 接口 / 目标
- 测试内容
- 结果
- 关联

不要把完整请求响应流水写进 `tests.md`。
