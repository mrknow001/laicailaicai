# 项目声明能力

## 项目声明面

项目声明是本项目的规则源，不是普通记录。开始任何目标访问、扫描、fuzz、PoC 验证、接口测试前，必须先建立或读取当前项目声明。

项目声明字段：

- 项目名称
- 测试账号
- 允许域名
- 允许 IP / IP 段
- 禁止域名
- 禁止 IP / IP 段
- 漏洞收录要求
- 注意事项
- 补充说明

默认项目文件位于 `projects/<项目标识>/project.md`。创建新项目时，复制 `projects/_template` 到 `projects/<项目标识>/`，再按模板填写。

补充说明用于纠偏和追加项目级规则，可以通过 `/update-note` 增加。补充说明可以加强约束、调整重点、降低风险，但不能绕过全局安全红线。

不要把自己推导出的请求上限、批次数量、测试次数等工程化约束写入项目声明。

## 默认漏洞收录要求

如果用户没有提供项目专属收录要求，默认优先关注高危和严重风险：

- 未授权访问
- 越权访问
- 敏感信息泄露
- 认证绕过
- 严重业务逻辑漏洞
- 可利用 SQL 注入的存在性证明
- 文件上传 / 文件下载风险
- RCE / 命令执行存在性证明
- 直接影响账号安全、系统权限、核心数据的漏洞

默认低优先级或不作为默认测试项：

- 普通 URL 跳转
- 普通反射 XSS
- 无敏感影响的信息泄露
- 低价值配置暴露
- 无法证明实际影响的轻微信息暴露

当项目声明只收高危或严重漏洞时，低优先级漏洞默认不作为测试项；如果顺手发现，只做简要记录，不偏离主线。

## 默认注意事项

如果用户没有提供项目专属注意事项，默认遵守：

- 不进行账号爆破、弱口令批量验证、密码喷洒或撞库。
- 禁止测试删除接口，只记录发现情况；除非用户明确允许具体接口、测试方式和停止条件。
- 禁止测试修改、支付、短信、邮件、批量导出接口；除非用户明确允许具体接口、测试方式和停止条件。
- 不进行高并发扫描、DoS 或影响稳定性的 fuzz。
- fuzz 必须低速、小字典、可熔断，并优先服务于高危漏洞发现。
- 非明显只读接口测试前必须说明接口来源、功能判断依据、预期影响，并询问用户。
- 对 Web / App / 小程序中提取到的接口，先判断后端域名或 IP 是否在允许范围内。
- 如果测试方向反复偏向低价值漏洞，必须回到漏洞收录要求重新聚焦。

## 默认补充说明

新项目默认写入：

- 不要为了覆盖率而无脑爆破或大范围 fuzz。
- 优先围绕接口权限、认证绕过、敏感数据、业务逻辑和高危入口构造漏洞假设。
- 删除、修改、支付、短信、邮件、批量导出等危险接口禁止测试，默认只进入“未测试危险接口清单”。
- 信息收集必须服务于渗透测试，不做无目标的资产堆积。
- 每次验证前先判断该测试是否符合漏洞收录要求和项目注意事项。
- 目标过多时分批执行，不要为了覆盖更多目标而降低单个目标的验证完整度。
