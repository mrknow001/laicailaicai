# 渗透流程与验证能力

## 渗透作战循环

本智能体采用 5 个工作面，而不是繁重状态机：

1. 项目声明面：定义项目规则、范围、禁区、收录偏好和补充约束。
2. 线索收集面：从 Web、App、小程序、搜索、抓包、历史经验中提取接口、参数、认证、业务和技术栈线索。
3. 漏洞假设面：围绕线索提出可能的高价值漏洞假设，并按收录要求排序。
4. 最小验证面：完成当前漏洞假设所需的最小完整验证闭环，避免无关请求、无关参数和扩大化验证。
5. 沉淀输出面：记录测试、接口、证据、未测危险点、报告和经验。

## 最小化验证的正确含义

最小化验证不是“把请求数量压到最少”，而是“完成当前漏洞假设所需的最小完整验证闭环”。

必须区分：

- 应该减少：无关参数、无关接口、无关漏洞类型、大字典、高并发、重复请求、为了碰运气乱打 PoC。
- 不应减少：当前特长、当前漏洞假设或当前验证方法定义的核心检查点。

如果一个特长定义了核心变体、检查点或最小动作集合，这些内容就是该特长的最小完整验证闭环。不能为了“少请求”只抽样执行其中一部分。

目标很多时，正确做法是：

- 分批处理目标。
- 降低本轮目标数量。
- 记录部分测试状态。
- 暂停并询问用户是否继续。

错误做法是：

- 每个目标只象征性测一次。
- 跳过特长核心检查点后仍标记“已测无发现”。
- 自行发明请求上限并写入项目声明。

如果无法完成最小完整验证闭环，只能标记为“部分测试”或“未完成”，不能标记为“已测无发现”。

## L1 / L2 / L3 渗透深度策略

L1、L2、L3 是渗透测试深度和测试强度策略，不是漏洞类型，也不是固定流程。具体测试动作仍由项目声明、漏洞假设、特长和用户指令共同决定。

默认使用 L2。除非用户明确指定 L1 / L3，或项目注意事项要求更低影响，否则按 L2 的人工渗透思路工作。

通用规则：

- 任意层级都必须先读取项目声明并完成 scope check。
- 禁止范围、注意事项、补充说明和漏洞收录要求优先于层级策略。
- 任意层级都不能跳过特长、漏洞假设或验证方法定义的核心检查点。
- 任意层级遇到限流、封禁、验证码升级、服务异常、响应异常变慢或业务风险升高时，立即停止并询问用户。
- 目标过多时缩小批次，不要为了覆盖更多目标而降低单个目标的验证完整度。

### L1：快速低影响

L1 用于快速发现接口、敏感信息和高价值入口，适合 SRC 初筛、多目标初筛和测试窗口有限的场景。

默认信息收集：

1. 使用 `/info-find` 对本地文件、前端源码、JS、App / 小程序解包内容或用户提供材料做快速信息提取。
2. 围绕 `/info-find` 提取内容快速审计 JS，了解接口、参数、BaseURL、认证线索、功能模块和其他相关信息。
3. 快速访问网站，判断站点用途、主要功能、登录入口、公开接口和明显业务模块。
4. 将已发现接口与 BaseURL 拼接后快速访问：先 GET；如果提示方法不对，再尝试 POST。
5. 按访问结果归类：GET 可访问、POST 可访问、GET / POST 都不可访问、403、404、405、50x、其他异常。

L1 默认测试项：

- 未授权访问。
- 信息泄露。
- SQL 注入存在性证明，仅做闭合和排序测试。
- 使用特有小字典做轻量参数 fuzz。

L1 额外测试：

- 默认测试项以外的漏洞类型，必须先和用户确认测试方案、影响范围和停止条件。
- 不做大字典、大范围 fuzz、账号爆破、验证码绕过、密码喷洒或危险接口测试。
- 删除、修改、支付、短信、邮件、批量导出等危险接口禁止测试；除非用户在项目声明或补充说明中明确允许具体接口、测试方式和停止条件。

L1 输出：

- 标记为 `L1`。
- 记录接口归类、默认测试项结果、未完成范围和是否建议进入 L2。

### L2：模拟人工渗透

L2 是默认档位。核心不是“更猛”，而是像人类渗透测试一样先理解业务、建立假设，再验证。

信息收集：

- 信息收集要比 L1 更全面，挖掘过程中持续补充 `leads.md`、`interfaces.md`、`hypotheses.md` 和 `tests.md`。
- 审计数据包、JS、接口命名、参数语义、返回结构、角色边界和业务状态。
- 对一组业务流程涉及的多个数据包，要串起来理解原本设计逻辑。

验证思路：

- 拿到接口或接口组后，先预测漏洞概率，再决定是否验证。
- 不要上来就把 SQL 注入、XSS、SSRF、文件上传等通用测试全部打一遍。
- 对参数、接口、枚举值可以做简单模糊测试，但猜不出来就记录下来，不无止境尝试。
- 生成针对性字典时控制在 2000 条以内；有精简字典时优先使用精简字典。
- 连续多个同类接口验证同类漏洞都无发现时，应重新评估下一个同类接口是否还有必要测试该方向。
- 对 GET 返回 500、参数缺失、方法不确定等情况，不要简单按失败处理；应思考是方法不对、参数不对、服务端异常还是触发了有效线索。

回头看机制：

- 对“只差一个参数、ID、状态值、枚举值、token 或上下文”的候选漏洞，记录为回头看项。
- 后续如果在其他 JS、接口、响应包或业务流程中遇到相关值，必须回头复测。
- 人类挖洞不是一个数据包测完就永远放弃；L2 必须保留这种跨接口、跨流程的回看能力。
- 回头看项也要给用户阅读。必须记录为什么觉得该接口 / 接口组值得回头看，当前缺什么，以及用户可以补充什么信息。
- 回头看项优先写入 `hypotheses.md`，并在 `tests.md` 中记录触发来源和复测结果。

L2 输出：

- 标记为 `L2`。
- 记录业务理解、漏洞假设、验证动作、结果、回头看项和需要用户确认的问题。

### L3：深度白名单

L3 是 L2 的深度补充。L2 通常是“可疑才继续”，L3 可以对“有可能”的方向做更完整覆盖，但必须受控。

启用条件：

- 用户明确要求 L3 或深度测试。
- 已明确白名单资产、路径、接口、模块或业务流程。
- 已明确字典规模、请求速度、并发限制、时间窗口和熔断条件。

允许方向：

- 更深的接口猜测。
- 更深的参数猜测。
- 更深的枚举值猜测。
- 对 L2 回头看项做集中复测。
- 在白名单内做受控 fuzz，发现更多潜在漏洞。

张弛有度：

- L3 不是无脑测试；明显低概率、低价值的位置不测，例如在 `pageSize` 这类参数上测试 XSS。
- 不对同类接口反复做无意义 fuzz；连续低收益时应收缩方向。
- 字典优先使用精简版本；针对性字典默认控制在 2000 条以内，除非用户明确允许更大规模。
- 发现风控、限流、响应变慢、错误率升高或业务异常时立即停止。

禁止动作：

- 不跨出白名单。
- 不使用破坏性 payload。
- 不进行高并发、DoS、持久化控制、提权或内网横向。
- 删除、修改、支付、短信、邮件、批量导出等危险接口禁止测试；除非用户在项目声明或补充说明中明确允许具体接口、测试方式和停止条件。

L3 输出：

- 标记为 `L3`。
- 记录白名单、字典来源、强度限制、熔断条件、批次结果和未完成范围。
- 未覆盖范围必须标记“未完成”或“未测试”，不能写成“已测无发现”。

### 层级切换

- 用户未指定时默认 L2。
- 项目影响容忍度低、目标很多、仅需快速初筛时，使用 L1。
- L1 发现高价值入口后，建议升级 L2。
- L2 需要更完整覆盖，且用户明确白名单和强度限制时，升级 L3。
- 出现范围不清、影响不明、风控或业务异常时，降级或停止询问用户。
