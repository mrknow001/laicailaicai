# 渗透流程与验证能力

## 渗透作战循环

本智能体采用 5 个工作面，而不是繁重状态机：

1. 项目声明面：定义项目规则、范围、禁区、收录偏好和补充约束。
2. 线索收集面：从 Web、App、小程序、搜索、抓包、历史经验中提取接口、参数、认证、业务和技术栈线索。
3. 漏洞假设面：围绕线索提出可能的高价值漏洞假设，并按收录要求排序。
4. 最小验证面：完成当前漏洞假设所需的最小完整验证闭环，避免无关请求、无关参数和扩大化验证。
5. 沉淀输出面：记录测试、接口、证据、未测危险点、报告和经验。

## 最小化验证的正确含义

最小化验证不是“把请求数量压到最少”，而是“完成当前漏洞假设所需的最小完整验证闭环”。

必须区分：

- 应该减少：无关参数、无关接口、无关漏洞类型、大字典、高并发、重复请求、为了碰运气乱打 PoC。
- 不应减少：当前特长、当前漏洞假设或当前验证方法定义的核心检查点。

如果一个特长定义了核心变体、检查点或最小动作集合，这些内容就是该特长的最小完整验证闭环。不能为了“少请求”只抽样执行其中一部分。

目标很多时，正确做法是：

- 分批处理目标。
- 降低本轮目标数量。
- 记录部分测试状态。
- 暂停并询问用户是否继续。

错误做法是：

- 每个目标只象征性测一次。
- 跳过特长核心检查点后仍标记“已测无发现”。
- 自行发明请求上限并写入项目声明。

如果无法完成最小完整验证闭环，只能标记为“部分测试”或“未完成”，不能标记为“已测无发现”。

## L1 / L2 / L3 策略坑位

L1、L2、L3 是核心渗透深度策略，但当前尚未精细化落地。

在策略未完善前：

- `/deep-l1`、`/deep-l2`、`/deep-l3` 只作为预留入口。
- 不要自行扩写三种策略的具体测试动作。
- 不要因为用户调用策略命令就自动扫描或 fuzz。
- 只能围绕项目声明、线索和风险边界进行讨论、准备和记录。

后续策略落地时，应遵守这个定位：

- L1：快速、低频、只读、少无关请求，适合 SRC 快速找高价值漏洞。
- L2：模拟人工渗透，重逻辑判断，非明显查询接口需要用户确认。
- L3：深度白名单测试，可使用受限 fuzz，但必须明确白名单、并发、字典规模、时间窗口和熔断条件。
