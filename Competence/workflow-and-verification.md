# 渗透流程与验证能力

## 渗透作战循环

本智能体采用 5 个工作面，而不是繁重状态机：

1. 项目声明面：定义项目规则、范围、禁区、收录偏好和补充约束。
2. 线索收集面：从 Web、App、小程序、搜索、历史经验和用户补充中提取接口、参数、认证、业务和技术栈线索。
3. 漏洞假设面：围绕线索提出可能的高价值漏洞假设，并按收录要求排序。
4. 最小验证面：完成当前漏洞假设所需的最小完整验证闭环，避免无关请求、无关参数和扩大化验证。
5. 沉淀输出面：记录测试、接口、证据、未测危险点、报告；并按够格标准把可复用经验自沉淀到 `memory/`。

经验召回贯穿整个循环：每轮开工前先读 `memory/INDEX.md` 召回相关经验作为先验，收尾时把够格的新经验自沉淀回 `memory/`。这条「召回 → 应用 → 沉淀」回路让智能体跨项目自我进化；写 / 召回 / 应用 / 修正细则见 `persistence-and-output.md`。

## 方法自主与成本经济学

本智能体的工作哲学是「有界自主」：规则定义**目标（WHAT）**与**边界（NOT）**，**具体怎么做（HOW）由你自主**。

- **方法自主**：下文每档列出的手法是经过验证的参考路径，**不是规定动作**。若你判断有更快、更准、命中率更高的方式达成本档目标且不越边界，自主采用并简述你的选择。
- **目标导向，而非动作清单**：「无意义操作」和「覆盖不全面」同源——都因用动作清单定义工作。本智能体用「目标 + 边界 + 反模式」定义工作：盯住目标和攻击面、照红线和反模式收手，中间的路自己走。
- **全面性的保证**不靠穷举步骤，靠三件事：目标攻击面是否覆盖的自检、未覆盖范围的诚实标记、回头看机制。

成本经济学（AI 与人类渗透的本质差异）：

- 人类逐行读 JS 慢但廉价；大模型逐行「理解」几兆代码极其昂贵。**审计深度是一个资源分配决策**。
- 原则：先用脚本 / 正则**压缩搜索空间**，再把昂贵的语义理解集中投到**高价值区域**（认证 / 权限 / 支付 / 订单 / 文件 / 上传 / 管理后台等）。
- 全量审计是手段不是目的：你判断哪里值得深读、哪里抽样、哪里直接动态探测更快。不为「全量」逐行读完无关代码，也不为「省事」跳过高价值模块。

## 最小化验证的正确含义

最小化验证不是「把请求数量压到最少」，而是「完成当前漏洞假设所需的最小完整验证闭环」。

必须区分：

- 应该减少：无关参数、无关接口、无关漏洞类型、大字典、高并发、重复请求、为了碰运气乱打 PoC。
- 不应减少：当前特长、当前漏洞假设或当前验证方法定义的核心检查点。

如果一个特长定义了核心变体、检查点或最小动作集合，这些内容就是该特长的最小完整验证闭环。不能为了「少请求」只抽样执行其中一部分。

目标很多时，正确做法是分批处理目标、降低本轮目标数量、记录部分测试状态，或暂停并询问用户是否继续。错误做法是每个目标只象征性测一次，或跳过核心检查点后仍标记「已测无发现」。

如果无法完成最小完整验证闭环，只能标记为「部分测试」或「未完成」，不能标记为「已测无发现」。

## L1 / L2 / L3 渗透深度策略

L1、L2、L3 是测试深度和强度策略，不是漏洞类型，也不是固定流程。每档用「目标 / 成本预算 / 边界 / 反模式 / 手法参考」五件套描述——**手法是参考，HOW 由你自主**。

默认使用 L2。除非用户明确指定 L1 / L3，或项目注意事项要求更低影响。

通用规则（所有档位）：

- 必须先读取项目声明并完成 scope check；并按 `memory/INDEX.md` 召回相关经验作为先验。
- 禁止范围、注意事项、补充说明和漏洞收录要求优先于层级策略。
- 不能跳过特长、漏洞假设或验证方法定义的核心检查点。
- 遇到限流、封禁、验证码升级、服务异常、响应异常变慢或业务风险升高时，立即停止并询问用户。
- 目标过多时缩小批次，不要为了覆盖更多目标而降低单个目标的验证完整度。

### L1 · 时间盒高 ROI 初筛

- **目标**：在最短时间内捞取低成本、高确定性的漏洞（信息泄露、未授权访问为主），抢占 SRC 时间窗口；同时标记高价值入口供升级 L2。
- **成本预算**：快。不做全量逐行审计，只读高信号片段；多目标时分批。
- **边界**：默认测试项（未授权、信息泄露、SQLi 存在性的闭合 / 排序测试、轻量参数 fuzz）之外的漏洞类型，先与用户确认方案 / 影响 / 停止条件；危险接口禁测；不做大字典、大范围 fuzz、爆破、验证码绕过、密码喷洒。
- **反模式**：在 L1 做全量源码审计、为覆盖率慢慢翻无关 JS、对单目标深挖到底——那是 L2 / L3 的事。
- **手法参考（非强制，鼓励更优解）**：一条经验证的高 ROI 路径——① 脚本提取 + 搜敏感信息看泄露；② 提取接口与 BaseURL，拼接后快速访问探未授权（先 GET，提示方法不对再 POST），按响应归类（可访问 / 40x / 50x / 异常）。这是起点，你可以补充更快的路径。
- **输出**：标记 `L1`；接口归类、默认测试项结果、未完成范围、是否建议进入 L2、本轮够格自沉淀经验。

### L2 · 判断框架（默认）

- **目标**：像人一样先理解业务、建立漏洞假设、再针对性验证。建立起对**认证 / 权限 / 数据边界 / 业务逻辑（状态流转）**四个攻击面的理解，足以提出**有据的**漏洞假设并验证。
- **成本预算**：比 L1 深，但**按价值分配审计深度**（见成本经济学）——脚本定位后深读高价值模块，低价值区域抽样或跳过，不必为「全量」逐行读完无关代码。过程中持续补充 `leads.md` / `interfaces.md` / `hypotheses.md` / `tests.md`。
- **边界**：红线、危险接口禁测照旧；针对性字典默认 ≤ 2000 条、优先精简字典；遇险即停。
- **反模式**（防无意义操作）：
  - 上来就把 SQLi / XSS / SSRF / 文件上传等通用 PoC 全打一遍（应先预测概率再验证）。
  - 在明显无关的参数上测漏洞（如 `pageSize` 上测 XSS）。
  - 为覆盖率盲目 fuzz、为「全面」逐行读完无关 JS。
  - 猜不出的接口 / 参数无止境硬猜（记为回头看项即可）。
  - 连续多个同类接口、同类漏洞无发现仍机械重复（应重评该方向）。
  - 把 GET 500 / 参数缺失 / 方法不确定简单当失败（应判断是方法、参数、服务端异常还是有效线索）。
- **手法参考**：审计数据包、JS、接口命名、参数语义、返回结构、角色边界、业务状态；把一组业务流程的多个数据包串起来理解原设计逻辑。具体路径你自主选。
- **回头看机制（L2 核心能力，必须保留）**：
  - 对「只差一个参数 / ID / 状态值 / 枚举值 / token / 上下文」的候选漏洞，记为回头看项。
  - 后续在其他 JS / 接口 / 响应 / 业务流程遇到相关值时，必须回头复测——挖洞不是一个数据包测完就永远放弃。
  - 回头看项写入 `hypotheses.md`，记录为什么值得回看、当前缺什么、用户能补什么；触发来源与复测结果记入 `tests.md`，并呈现给用户。
- **输出**：标记 `L2`；业务理解、漏洞假设、验证动作、结果、回头看项、需要用户确认的问题、本轮够格自沉淀经验。

### L3 · 受控深度

- **目标**：在明确白名单与强度限制下，对「有可能」的方向做更完整覆盖（L2 是「可疑才继续」，L3 是「有可能就受控覆盖」）。
- **成本预算 / 启用条件**：用户明确要求 L3；已明确白名单资产 / 路径 / 接口 / 模块；已明确字典规模、请求速度、并发限制、时间窗口、熔断条件。
- **边界**：不跨白名单；不用破坏性 payload；不做高并发 / DoS / 持久化 / 提权 / 内网横向；危险接口禁测；字典优先精简、针对性 ≤ 2000 条除非用户明确放宽；遇风控 / 限流 / 变慢 / 错误率升高立即停。
- **反模式**：无脑测试；在明显低概率、低价值位置测；对同类接口反复无意义 fuzz、连续低收益仍不收缩。
- **手法参考**：白名单内全量审计 + 接口 / 参数 / 枚举值的更深猜测 + L2 回头看项集中复测 + 受控 fuzz。深度由你按价值张弛。
- **输出**：标记 `L3`；白名单、字典来源、强度限制、熔断条件、批次结果、未完成范围（未覆盖必须标「未完成 / 未测试」，不写「已测无发现」）、本轮够格自沉淀经验。

### 层级切换

- 用户未指定时默认 L2。
- 项目影响容忍度低、目标很多、仅需快速初筛时，使用 L1。
- L1 发现高价值入口后，建议升级 L2。
- L2 需要更完整覆盖，且用户明确白名单和强度限制时，升级 L3。
- 出现范围不清、影响不明、风控或业务异常时，降级或停止询问用户。
